## 무엇을 (사용자 결정 = Option B)

@here/@channel 을 @everyone(MENTION_EVERYONE·OWNER/ADMIN)과 **분리된 새 권한 비트
`MENTION_CHANNEL`(0x2000·기본 MEMBER 허용)**로 게이트하고, 대규모 범위 멘션에 **서버 측
임계값(@everyone≥6·@here/@channel≥50)을 enforce**한다(클라 confirm 의 안전망). **DB 마이그레이션 없음**
(base=코드 role-enum 맵·override 기존 Int 컬럼·bit13 unset=기본 ON).

## 설계 핵심

- **권한 비트**: `permissions.ts` `MENTION_CHANNEL=1n<<13n` + `CHANNEL_OVERWRITE_FLAGS`(14). base
  `MENTION_CHANNEL_ROLE_BASE`(OWNER/ADMIN/MODERATOR/MEMBER on·GUEST off). `resolveMentionScopes`
  (한 override findMany 로 두 비트 분리 fold·resolveMentionEveryone 은 위임). gate 분리
  (@everyone=hasMentionEveryone·@here/@channel=hasMentionChannel).
- **임계값 enforce**: send() + update() 둘 다. send 는 게이트 통과 broad 멘션, update 는 **편집으로
  새로 추가된** broad 멘션(이전 버전 delta)에 한해. 워크스페이스 멤버수(=broad fanout 실제 폭) ≥
  임계값 + `!bulkMentionConfirmed` → 409 `BULK_MENTION_CONFIRM_REQUIRED`(INSERT/UPDATE 전 throw·
  idempotency 미소비). 필터가 details{mention,count,threshold} surface.
- **override 검증**: `channels.controller` USER/ROLE override 가 `isWithinChannelOverrideBits`(BigInt
  subset·집행 0x17F + 멘션 0x80/0x2000)로 검증 — 종전 numeric `> ALL_PERMISSIONS` 가 MENTION_CHANNEL
  (0x2000>0x17F)을 거부하던 것 교정 + ADMINISTRATOR/모더레이션/예약 비트는 여전히 거부(int32 wrap 회피).
- **FE**: SpecialMentionConfirmDialog @channel 카피·specialMention 'channel' 키·MessageComposer/
  MessageList 가 send/edit 409 → 같은 nonce 재전송(MED-1: 낙관행 고아 제거)·permissionCatalog 노출.
